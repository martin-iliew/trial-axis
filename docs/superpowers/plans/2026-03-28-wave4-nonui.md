## Task 1: Write SQL Integrity Assertions

**Files:**
- Create: `web/supabase/assert_seed.sql`

- [ ] **Step 1: Create the assertions file**

Create `web/supabase/assert_seed.sql` with the following content. Each query returns zero rows when healthy and non-zero rows when broken:

```sql
-- ============================================================
-- Seed Integrity Assertions
-- Run against the cloud Supabase project after seeding.
-- Every query must return 0 rows for the seed to be valid.
-- ============================================================

-- 1. Clinics missing certifications
SELECT c.id, c.name, 'missing certifications' AS issue
FROM clinics c
LEFT JOIN certifications cert ON cert.clinic_id = c.id
WHERE cert.id IS NULL;

-- 2. Clinics missing equipment
SELECT c.id, c.name, 'missing equipment' AS issue
FROM clinics c
LEFT JOIN equipment e ON e.clinic_id = c.id
WHERE e.id IS NULL;

-- 3. Clinics missing specializations
SELECT c.id, c.name, 'missing specializations' AS issue
FROM clinics c
LEFT JOIN clinic_specializations cs ON cs.clinic_id = c.id
WHERE cs.therapeutic_area_id IS NULL;

-- 4. Match result variation broken (all rank-1 results point to same clinic)
-- Run AFTER executing /api/match for all 3 seed trial projects.
SELECT 'broken_match_variation' AS issue, COUNT(DISTINCT top.clinic_id) AS unique_winners
FROM (
  SELECT DISTINCT ON (trial_project_id)
    trial_project_id,
    clinic_id
  FROM match_results
  ORDER BY trial_project_id, score DESC
) top
HAVING COUNT(DISTINCT top.clinic_id) = 1;

-- 5. Demo accounts with wrong or missing role
SELECT id, role, 'wrong or missing role' AS issue
FROM profiles
WHERE role IS NULL OR role NOT IN ('sponsor', 'clinic_admin');
```

- [ ] **Step 2: Run assertions 1–3 and 5 in the Supabase dashboard**

Open the Supabase cloud project → SQL Editor. Paste and run each query individually (assertions 1, 2, 3, and 5 — skip #4 until match results exist).

Expected: each returns 0 rows. If any return rows, open `web/supabase/seed.sql`, find the clinic(s) listed in the results, and add the missing rows (certifications, equipment, or specializations). Then run `npx supabase db reset` and re-run the assertions before continuing.

- [ ] **Step 3: Commit**

```bash
git add web/supabase/assert_seed.sql
git commit -m "chore(db): add seed integrity assertion queries"
```

## Task 3: Build Validation (DB Dev)

**Files:**
- Modify: any files with TypeScript or build errors (unknown until build runs)

- [ ] **Step 1: Run the build locally**

```bash
cd web && npm run build
```

Expected: build completes with no errors. If there are errors, proceed to Step 2.

- [ ] **Step 2: Fix TypeScript errors**

Common failure patterns and fixes:

**Missing Supabase types** — if `types/supabase.ts` is a stub, regenerate:
```bash
npx supabase gen types typescript --linked > types/supabase.ts
```

**`any` type usage** — replace with the correct Supabase-generated type. Example:
```ts
// before
const clinic: any = data

// after
import type { Tables } from '@/types/supabase'
const clinic: Tables<'clinics'> = data
```

**Server/client boundary violation** — if a Server Component imports from `lib/supabase/client.ts`, change the import to `lib/supabase/server.ts`.

**Missing `await` on `createServerClient()`** — the server client is async:
```ts
// before
const supabase = createServerClient()

// after
const supabase = await createServerClient()
```

- [ ] **Step 3: Re-run build until clean**

```bash
npm run build
```

Expected: exits with code 0, no red error lines.

- [ ] **Step 4: Commit all fixes**

```bash
git add -p   # stage only the fix changes
git commit -m "fix(build): resolve TypeScript errors before deploy"
```
